# Contributing to PingUp 🟢

First off, thank you for considering contributing to PingUp! It's people like you that make PingUp such a great tool for everyone.

### 🚀 Getting Started

1. **Fork the Repository**: Create your own copy of the project.
2. **Clone the Repository**: 
   ```bash
   git clone https://github.com/sabeenaviklar/PingUp.git
   ```
3. **Create a Branch**: 
   ```bash
   git checkout -b feature/YourFeatureName
   ```
4. **Make Your Changes**: Implement your feature or fix.
5. **Commit Your Changes**: 
   ```bash
   git commit -m 'Add some feature'
   ```
6. **Push to the Branch**: 
   ```bash
   git push origin feature/YourFeatureName
   ```
7. **Open a Pull Request**: Submit your changes for review.

### 🐛 How to Report a Bug

If you find a bug, please help us by providing as much information as possible:
- **Use a clear and descriptive title** for the issue.
- **Describe the exact steps** which reproduce the problem.
- **Explain which behavior you expected** and what you saw instead.
- **Include screenshots or GIFs** if the bug is related to the UI.
- **Mention your OS and browser version**.

### 💡 How to Suggest a Feature

We are always open to new ideas! To suggest a feature:
1. Check if the feature has already been suggested in the **Issues** tab.
2. Open a new issue and use the **Feature Request** label.
3. Provide a clear description of the feature and why it would be useful.

### 📝 Coding Standards

To maintain a clean and consistent codebase, please follow these guidelines:
- **JavaScript**: We use ES6+ syntax. Follow the AirBnB style guide where possible.
- **CSS**: Use the existing CSS variables for colors and spacing. Ensure your styles are responsive.
- **Components**: Keep components modular and reusable.
- **Comments**: Write clear, concise comments for complex logic.

### 📜 Commit Conventions

We follow a basic version of **Conventional Commits**:
- `feat:` for new features
- `fix:` for bug fixes
- `docs:` for documentation changes
- `style:` for formatting, missing semi-colons, etc.
- `refactor:` for code changes that neither fix a bug nor add a feature
- `chore:` for updating build tasks, package manager configs, etc.

Example: `git commit -m "feat: add emoji support to message input"`

### 🏁 Pull Request Process

1. Ensure any install or build dependencies are removed before the end of the layer when doing a build.
2. Update the README.md with details of changes to the interface, this includes new environment variables, exposed ports, and useful file locations.
3. You may merge the Pull Request once you have the sign-off of two other developers, or if you do not have permission to do so, a maintainer will review and merge it for you.

### 🌟 GSSoC 2026

If you are a **GSSoC** participant:
- Comment on the issue you want to work on.
- Wait for the issue to be assigned to you before starting work.
- Include your GSSoC ID in your PR description.
